# v0.3 Runner Design

Status: implemented on `feat/v0.3-runner` after the v0.2.4
self-registration/self-poll fixes. This is a shipping design artifact for the
v0.3 runner and launcher shims.

## Goal

`agentchute run` is the v0.3 reference wake adapter for non-tmux agents. It owns
the wrapper process, keeps registration and heartbeat state fresh, polls the
recipient inbox, and injects an inbox-check prompt into the real interactive
session when mail arrives.

The protocol does not change: senders deliver to a shared inbox, recipients are
responsible for reading it, and wake remains an implementation detail.

## Decisions

1. Add `agentchute run --as <id> --vendor <vendor> -- <wrapper> [args...]`.

   The runner launches the wrapper as a child process under a PTY, mirrors
   terminal I/O, forwards resize/signals, and owns a small control socket. The
   preferred socket path is `.agentchute/loop/state/<agent>/runner.sock`, but
   implementations may use a short path under the OS temp directory when the
   repo-local path exceeds Unix socket path limits. `runner.json` records the
   actual socket path.

2. Make normal wrapper commands route through the runner after install/init.

   `agentchute shims install` enables launcher shims for the known wrappers;
   the installer runs it by default:

   - `claude` / `claude-code` -> `agent_id: claude-code`, `vendor: anthropic`
   - `codex` -> `agent_id: codex`, `vendor: openai`
   - `gemini` / `gemini-cli` -> `agent_id: gemini-cli`, `vendor: google`

   The installer owns a user-level shim directory, defaulting to
   `~/.agentchute/bin`, and adds shell-profile PATH entries when the shim dir or
   binary dir is missing from `PATH` (operators can opt out with
   `AGENTCHUTE_NO_PATH_UPDATE=1`). Each shim discovers the current control repo.
   If the cwd is not inside an initialized agentchute pool, it execs the real
   wrapper unchanged. If the cwd is inside a pool, it execs `agentchute run`
   with the canonical agent id and the real wrapper path.

   This is what makes the user's normal command work after install/init. A
   repo-local runner that only works when the user remembers to type
   `agentchute run ...` is not sufficient for v0.3.

3. Use a real PTY, not named pipes or terminal-device writes.

   Interactive wrappers need a controlling terminal, line discipline, terminal
   modes, window size, and normal Ctrl-C behavior. A pipe-only launcher will be
   fragile for TUIs. Direct writes to another terminal remain out of scope as a
   default because they are hard to authorize and easy to misroute.

4. Prefer `github.com/creack/pty` for the first implementation.

   This is a deliberate dependency exception. The alternative is raw
   platform-specific PTY code for macOS/Linux or shelling out to `script` or
   `expect`. Raw PTY code increases bug surface in exactly the area where v0.3
   needs reliability; shelling out adds portability and quoting failure modes.
   Keep the dependency isolated behind `internal/runner/pty` so it can be
   replaced later if needed.

5. Polling remains the core inbox trigger.

   The runner polls inbox metadata on a short interval and updates heartbeat
   state on every tick. Do not make `fsnotify` part of the core path. Shared
   inboxes may live on network mounts where OS file notifications are absent or
   unreliable.

6. Register the runner, not a stale terminal pane.

   While running, registration should use:

   - `wake_method: agentchute-run`
   - `wake_target: unix:<absolute runner socket path>`
   - `host: <current host>`
   - `last_seen: <updated each poll tick>`
   - `status: active`

   On child exit, the runner marks the registration `offline` and removes its
   socket. `doctor` treats missing runner sockets as stale wake targets, and a
   future runner start refuses live collisions while cleaning up stale runner
   state.

   `doctor` and `gate` should treat a reachable `agentchute-run` socket as a
   reachable wake target, equivalent to a live tmux pane. A runner-owned agent
   does not need a separate `state/<agent>/poller.json` heartbeat because the
   runner already advances `last_seen` and exposes a local wake socket. Hook
   templates may still call `poller ensure`; it no-ops when a reachable
   `agentchute-run` target is registered.

7. Refuse multi-runner collisions by default.

   If `agentchute run --as codex` starts while another live runner already owns
   `codex`, the second runner refuses with a clear error naming the existing
   PID and socket path. Stale state is recoverable: if the PID is gone or the
   socket is unreachable, the new runner may clean up and start. A future
   `--takeover` flag can be added if operators want explicit replacement, but
   silent takeover is not the default.

8. Sender-side wake is only an accelerator.

   The `agentchute-run` wake adapter connects to the recipient's local Unix
   socket and sends a wake request. If the host differs, the socket is missing,
   or the wake fails, delivery still succeeds and the runner's own polling loop
   should pick up the inbox entry.

9. Injection policy is explicit.

   Default behavior:

   - If there is unread inbox mail and no injection is pending, enqueue one wake.
   - If the wrapper is confidently at an idle prompt, inject a minimal prompt
     such as `check inbox`.
   - If the wrapper is busy, queue the wake and inject when it returns to an
     idle prompt.

   Default interrupt policy is `--interrupt-policy=after-idle`. This protects
   in-flight model output and tool work. More aggressive modes are available
   for operators who prefer lower latency:

   - `after-idle`: queue while busy; inject only at a prompt. Default.
   - `after-grace`: queue while busy, then send Ctrl-C after a grace period and
     inject.
   - `always`: send Ctrl-C before injection whenever the runner is not
     confident the prompt is idle.

   This keeps the useful lesson from Aider watch mode (external file events can
   wake the interactive loop) without treating all inbox mail as permission to
   cancel the user's current turn.

10. Codex app-server is an optional Codex provider, not the universal default.

   Prototype result on 2026-05-21:

   - Started `codex app-server --listen ws://127.0.0.1:47321`.
   - Opened a Codex TUI with
     `codex --remote ws://127.0.0.1:47321 --disable hooks --no-alt-screen`.
   - Sent `turn/start` from a separate WebSocket client to the TUI's loaded
     thread.
   - The live TUI displayed the injected user prompt and streamed the response
     `AGENTCHUTE_APP_SERVER_SYNC_OK`.

   Conclusion: app-server can be a lighter Codex-specific wake provider when
   Codex is launched against that app-server. This does not help a normal Codex
   TUI that was started without `--remote`, and it does not help Claude Code or
   Gemini CLI. PTY remains the universal runner.

   The prototype used `--disable hooks` to avoid touching this repo's live hook
   state while testing. App-server mode should not depend on hook-driven
   liveness; the runner/provider owns registration, polling, and heartbeat. If
   Codex app-server hooks conflict with remote-mode operation, disable those
   hooks deliberately for the provider and rely on runner liveness instead.

## Runner State

Write runner-owned state to `.agentchute/loop/state/<agent>/runner.json`:

```json
{
  "agent_id": "codex",
  "runner_pid": 12345,
  "child_pid": 12346,
  "socket_path": "/repo/.agentchute/loop/state/codex/runner.sock",
  "started_at": "2026-05-21T02:30:00Z",
  "last_poll": "2026-05-21T02:30:05Z",
  "last_injection": "2026-05-21T02:30:05Z",
  "pending_wake": false,
  "status": "active"
}
```

The heartbeat source of truth is still the registration `last_seen`; the state
file is diagnostic and recovery metadata.

## Socket Protocol

Use newline-delimited JSON on a local Unix socket:

```json
{"op":"wake","reason":"new_mail","message_count":1}
{"op":"status"}
{"op":"shutdown"}
```

The socket is local-only and created with owner-only permissions. The wake
handler must not accept arbitrary text to inject; it only triggers the configured
check-inbox prompt.

## Implementation Plan

1. Add `internal/runner` with PTY process management, terminal resize handling,
   signal forwarding, poll loop, injection queue, and runner state writes.
2. Add the `run` command and `agentchute-run` wake adapter.
3. Extend `doctor`, liveness checks, and registration reconciliation to validate
   `agentchute-run` sockets the same way tmux panes are validated today.
4. Add installer launcher shims so normal wrapper commands route through the
   runner inside initialized pools and pass through elsewhere.
5. Add runner collision checks and stale-state recovery.
6. Add integration tests with a fake interactive child process that records PTY
   input and simulates prompt/busy states.
7. Add a Codex app-server provider behind a separate command or flag only after
   the provider can launch Codex in remote mode and bind the created thread id.

## Acceptance Criteria

- Starting `agentchute run --as codex --vendor openai -- codex` creates a fresh
  registration with no tmux dependency.
- After install/init, running normal wrapper commands from an initialized repo
  routes through `agentchute run` without extra user typing; outside initialized
  repos the shims pass through to the real wrapper.
- New inbox mail is processed without human input while the runner is alive.
- `last_seen` advances even when no human turn occurs.
- Stale runner sockets are detected and cleared.
- A second live runner for the same `agent_id` refuses by default.
- Killing the child marks the registration offline.
- Existing tmux behavior and hook-managed poller behavior keep working.
- Verification passes: `gofmt -w .`, `go vet ./...`, `go test ./...`,
  `go build ./...`, and `sh tests/install_test.sh`.
