Running multiple AI agents on a shared project is easy.
Coordinating them isn't.

agentchute is a small coordination protocol: each agent owns an inbox; senders deliver identified messages without overwriting; recipients consume on their own cadence. Humans stay in the loop for decisions and steering — not for ferrying messages between terminals.

No central server or broker required. No SDK. No long-lived daemons required. Just AGENTCHUTE.md plus a short enrollment block your agents read and follow. 

curl -fsSL https://raw.githubusercontent.com/agentchute/agentchute/main/install.sh | sh

Or skip the binary — just drop AGENTCHUTE.md into your project and follow the enrollment block. The whole protocol fits in one file.

What's actually in the protocol

The protocol is a small set of implementation-agnostic primitives. Conforming implementations are free to choose any inbox medium, any transport between sender and inbox, and any wake mechanism — those are outside the protocol.

  • Per-recipient inbox. Each agent owns an ordered message stream. Senders deliver into the recipient's inbox; the recipient owns discovery and consumption.
  • Ordered, identified messages. Each carries a unique identity tuple; receivers process oldest-first by timestamp with deterministic tie-breaking.
  • No-overwrite delivery. Senders never silently clobbers an existing entry. Collisions retry with a fresh nonce.
  • Recipient-owned consumption. Only the recipient reads its own message bodies. Senders and sidecars use metadata only.
  • Recipient-side polling. The protocol's discovery mechanism is polling. Agents discover unread mail via their own inbox scans on their own cadence.
  • Optional wake. Senders MAY signal a recipient via an optional wake adapter (tmux, HTTP, SSH). Failure is logged and ignored; the protocol remains correct in its absence.

The reference implementation

The CLI in this repo picks one concrete answer for each implementation-specific axis. It maps the primitives onto Markdown files on a shared filesystem. While it ships with tmux wake support, v0.2 makes the **no-tmux** story first-class through a three-tier polling model.

  • Tier 1: Native loops. If your wrapper has a built-in recurring task feature (like Claude Code's /loop), use it. Zero infrastructure required.
  • Tier 2: Preflighted scheduler. For wrappers without a native loop, an external scheduler (cron, systemd, launchd) runs a side-effect-free preflight using agentchute self-poll and launches the wrapper only when work exists.
  • Tier 3: Lifecycle catchup. Active sessions catch new arrivals at lifecycle boundaries via hooks (e.g., Gemini's AfterAgent continuation).
  • Diagnostics: agentchute doctor provides pool-wide health checks for PATH, hooks, registrations, and loop scaffold. It can even --generate-service unit files to automate Tier 2.

The protocol is substrate-agnostic. The reference CLI uses a shared filesystem; another implementation can swap in a queue, HTTP endpoint, object store, or git transport. Agents can run on different machines, networks, or regions — same primitives either way.

Alternate inbox media and wake adapters are discussed in EXTENSIONS.md.
