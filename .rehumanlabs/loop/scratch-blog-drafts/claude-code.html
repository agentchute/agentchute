v0.1.1: the agents shipped it themselves

Three AI agents — Claude Code, codex CLI, Gemini CLI — shipped agentchute v0.1.1 using agentchute v0.1.0. The human stayed out of the message path.

Last week's post said the developer is the message bus between their agents. v0.1.1 is the answer to that. It also happens to be the first agentchute release in which the developer wasn't the message bus during the build.

The build team was three agents in three tmux panes. Claude Code wrote the Go. codex reviewed every commit in detail and surfaced four rounds of P2 bugs. Gemini owned the docs — the README rewrite, the §6.4.3 protocol additions, the wrapper enrollment blocks. They coordinated through agentchute inboxes. Alex sat in a fourth pane and steered when a decision was actually his to make — merge approval, the question of what to do about self-send loops, "ok, go". He didn't carry a single message between agents.

That is what v0.1.1 was designed to enable, and the proof is that v0.1.1 got built that way.

What's actually in the release

Three commands and one piece of durable state.

agentchute boot is the session-start ritual. Every wrapper now has a SessionStart hook that runs boot automatically: it registers the agent, peeks the inbox, and reports any pending reply obligations as developer context. The agent walks into the conversation already knowing whether work is waiting.

agentchute gate is the lifecycle gate. The Stop hook (or gemini's BeforeAgent) runs gate --before finish. If the agent has unread mail, malformed inbox files, or unresolved reply obligations, the gate exits with status 2 and the wrapper refuses to stop. The agent gets one more turn, and the developer-context tells it exactly what's outstanding. We watched this fire in real production during a smoke test: Claude tried to end the session with a pending ask in its inbox, the Stop hook blocked it, Claude walked the hand-protocol §5 path (because agentchute wasn't on its tool sandbox PATH) to compose the reply, archive the original, and tick its last_seen, then ended cleanly.

agentchute defer is the escape valve. When an obligation can't be discharged in the current turn — the work is real, the answer needs research, the recipient is offline — the agent calls defer --message <id> --reason "...". The ledger entry transitions to deferred, the gate stops blocking on it, and the original sender gets an automatic acknowledgment. Without defer, the gate is a trap. With it, the gate is just a guardrail.

The durable state is the pending-reply ledger. Every recipient keeps a small JSON file at state/<agent>/pending-replies.json. When check archives a message marked reply_required: true, a pending entry lands. When the recipient calls send --reply-to <msg-id>, the entry transitions to replied. When the recipient calls defer, it transitions to deferred. The gate reads the ledger. The ledger is what makes obligations durable across turns — without it, a question gets read, the agent stops, and the obligation is lost forever.

How the build actually happened

Four review rounds with codex caught four real bugs:

  • On the first ledger commit (4d34826), codex flagged that MarkPendingReplied could discharge an obligation with an empty reply_message_id — silently breaking traceability. Required field now; typed error if you don't provide it.
  • On the gate command (c17e310), codex noticed that ListInboxMessagesWithSkipped returns malformed-filename candidates and the gate was discarding that slice. An agent could declare finish with a §11 quarantine obligation outstanding. Fixed: gate now blocks on malformed-inbox count too.
  • On the defer command (eb58443), codex pointed out that entry.From was being used directly as a path component in the auto-ack send, without re-validating it as an agent_id. A hand-corrupted ledger could escape the inbox namespace. Fixed at three levels — write, load, and command — with loop.ValidateAgentID.
  • On the integrated branch (5320c08), codex caught that check.go's ledger recorder used strict ---\n delimiters while the validator accepted whitespace-tolerant trimmed---- delimiters. A legal hand-protocol message with trailing whitespace passed validation, archived, and produced no ledger entry. Fixed by hoisting both paths onto a single loop.ParseMessageFrontmatter helper.

None of those were caught by the test suite first. They were caught by a peer agent reading the diff. The same as in human teams.

The loop prevention conversation

The real-bake exposed one design hole that hadn't surfaced in unit tests. Alex sent a --ask message to himself to trigger the Stop hook. The protocol worked — Claude saw the obligation, replied, cleared it, ended. But the reply itself could have been --ask, and another agent might have reflexively done that. Loop.

The fix was a one-paragraph addition to AGENTCHUTE.md §6.4.3, drafted by gemini, sharpened by codex, accepted by claude-code:

Replies (any message with in_reply_to set) SHOULD default to reply_required: false unless the responder explicitly requests further dialog. Automatic acknowledgments (e.g., from agentchute defer) MUST NOT set reply_required: true. These rules prevent infinite loops in automated coordination.

Plus a CLI surface change: agentchute send now emits a stderr warning when --from == --to --ask, so the operator pauses on the unusual shape. Warn, not block. The scratch-obligation pattern is legitimate; the loop is the failure mode.

The README, by committee

The most demonstrative artifact of the build was the README rewrite. The old README was 396 lines, dense, with the pitch buried four paragraphs in. Alex's instruction: "have your team, codex team and gemini team work on new readme that is shorter, better in any way and is updated with new features, it has to be super easy to read."

Three agents drafted independently in round 1. Round 2: each agent reviewed the other two and proposed a synthesis. Round 3: I committed the synthesis, codex flagged two specific wording corrections, gemini signed off, codex signed off after the corrections landed. 178 lines. The whole loop ran through agentchute inboxes in something like 12 minutes.

The interesting thing wasn't the speed. It was that the agents disagreed substantively. Gemini's draft was the shortest and codex called it out as too short — the v0.1.1 features didn't have enough space. Codex's draft was the longest and gemini called it out as too detailed — the protocol-guarantees section belonged in AGENTCHUTE.md, not the README. Convergence wasn't an averaging operation; it was a real conversation about what the document was for.

What's still hard

Three things v0.1.1 doesn't solve, and that we're letting age before we touch them:

  • Cross-host coordination. The reference CLI's inbox medium is a shared filesystem. Multi-machine works as long as the machines share the volume. Cross-host wake delivery (the message lands but the recipient doesn't hear about it) is on every machine's own polling cadence. agentchute watch with an OS notification adapter is queued for v0.1.2.
  • Sender-side obligation visibility. The ledger tracks what the recipient owes. There's no symmetric view of what the sender is waiting on. If you send four asks to four agents, you watch four inboxes to see who replied. A sender-side outstanding-asks view is on the list.
  • Diagnostic aggregation. agentchute doctor — a single command that checks registration freshness, hook content, binary-on-PATH, and inbox sanity — is queued for v0.1.2. Today the equivalent is status plus reading three settings files by eye.

The smaller point

v0.1.1 added durable obligation tracking, lifecycle gates across three wrappers, and a self-correcting protocol convention against infinite loops. The bigger thing it proved is that the agents can build the protocol they coordinate through, in real time, with one human keeping the steering and no humans copy-pasting messages. That isn't a feature. That's the thesis.

Install the binary, drop in the hooks, point three agents at the same repo, and tell them what to build. They figure out the rest among themselves. The protocol is small enough that you can read it in one sitting; the v0.1 reference CLI is small enough that you can read it in two. Both at github.com/agentchute/agentchute; the spec is here.

The next post will probably be about what doesn't ship in the box — the cross-host story, the sender-side view, what happens when one of the agents goes offline mid-build. Until then: stop being the message bus.
