v0.1.2: doctor, watch, and the no-tmux story

Six days after v0.1.1 made the coordination work, v0.1.2 makes it diagnoseable. New: doctor, watch, a Claude-Code-specific hook output mode, and the first answer to "what if you don't have tmux?"

v0.1.1 made the loop work end-to-end. Three agents, durable reply obligations, lifecycle hooks across three wrappers, and the human kept their hands off the keyboard during the build. v0.1.2 is the smaller, flatter follow-up: the things you notice you need only after the loop is real.

What shipped

agentchute doctor is the diagnostic aggregator. It walks an ordered checklist — scaffold integrity, binary on PATH, hook content sanity, agent registration, ledger state, wake-target validity — and reports each with a severity tag: BLOCKER (exit nonzero), WARN (operational signal), SKIP (not applicable), or OK. Designed to be wired into a CI step or an operator script; v0.1.1's gate/boot own lifecycle enforcement, doctor diagnoses.

The most useful check is also the most boring: hook content sanity. v0.1.1 shipped a warning in the README — "never use agentchute check in a hook unless you're sure the output reaches the model's context." doctor now checks for it. It also catches the templated form (${AGENTCHUTE_BIN:-agentchute} check), the env-only form ($AGENTCHUTE_BIN check), and mixed files where one command is safe and another isn't. Codex caught all three bugs during the rolling review of the first doctor commit; the regex-rewrite to scan per-occurrence is in the v0.1.2 history.

agentchute watch is the recipient-side persistent watcher. v0.1.1's only wake adapter was tmux send-keys, which works beautifully when every agent is in a tmux pane and falls over the moment one isn't. Watch is the answer: poll your own inbox, fire one of three actions when new mail arrives.

  • --notify — OS notification. macOS osascript, Linux notify-send. Wakes the human, not the agent. Pair with whatever wrapper invocation you use to bring the model up.
  • --print — stdout line per new message. Pipe into anything.
  • --exec <cmd> — shell command per new message, with AGENTCHUTE_MSG_ID, AGENTCHUTE_FROM, and AGENTCHUTE_TASK as env vars. The only path that can wake an agent without tmux, IF the operator's automation knows how to relaunch the wrapper. Operator-owned; watch emits a startup stderr warning when --exec is enabled.

Watch is non-consuming. It never archives, never quarantines, never wakes peers. Mail still has to flow through check for the model to actually see it; watch is strictly the arrival-notification layer. It dedupes new mail by frontmatter message_id when present, filename otherwise — same identity convention the pending-reply ledger uses.

--claude-hook UserPromptSubmit is the small one. v0.1.1 shipped Claude Code's UserPromptSubmit hook as plain-text pending output, after a real-bake caught that the --json form silently failed to inject context. v0.1.2 adds the structured JSON path: hookSpecificOutput.additionalContext nested, exit 0 required, verified against the live wrapper docs. Convergent with the existing --codex-hook; the two emitters currently produce byte-identical output, with a test pinning that as a tripwire if they ever diverge.

Plus three smaller surface improvements: status now defaults to a no-agent pool overview when --as is omitted (pure UX nit, but a real one), Appendix B was ported into the official spec with the schema corrections from the v0.1.1 real-bake folded in, and EXTENSIONS.md now lists the OS notification adapters with their human-relay semantics labeled explicitly.

The no-tmux story

A surprising amount of agentchute discourse has been about tmux. The protocol doesn't require it. The v0.1 reference CLI ships exactly one wake adapter, and that adapter happens to be tmux send-keys. People kept hitting the same question: "what if I'm not running in tmux?"

The v0.1.1 answer was "your recipient needs a polling cadence; here are some wrapper-specific recurring-task patterns." That's still true, and now it's the second-best answer. The first-best answer is watch. Run it once per recipient pane (or once per recipient host), let it sit. New mail arriving in that recipient's inbox now fires a notification, prints to a pipe, or invokes a script — your choice. No tmux. No watchdog daemon.

The watch loop is intentionally boring. Polling, not fsnotify. Stdlib only — no new dependencies. Default cadence is 10 seconds; operator-tunable via --interval, minimum 1 second. The startup sweep captures the current inbox state as "already seen" so a relaunched watcher doesn't notify on backlog.

Codex's brainstorm note on this was the right shape: do not reopen the silent-drain confusion. Watch peeks; it never consumes. If your watcher fires a notification and the operator reads the message visually, the message is still in the inbox until the wrapper runs check. That separation — arrival surface vs. consume surface — is what the v0.1.1 hook warning was about, and watch honors it.

How the cycle ran (again)

Same shape as v0.1.1's build. Three agents in three panes; the human steered. Codex landed three review rounds across the v0.1.2 Go work and caught real bugs both times: the --json discovery-failure exit-0 in doctor (BLOCKER that the report said was a blocker but the exit code disagreed), the templated/env-only forms of the silent-drain antipattern, and a test-precision issue in the status UX patch where a nanosecond-anchored backdate masked the production last_seen update path.

Gemini drove the spec port: §B of the rev3 scratch document is now a real Appendix B in AGENTCHUTE.md, with the Claude Code schema corrected to the nested matcher + hooks shape and the --json example removed from the UserPromptSubmit slot. The §10/§12 notification contradictions surfaced as a follow-up review and got resolved in the same cycle.

What this looked like on the wire: one agentchute send --ask per commit Claude landed, one send --reply-to per fix, the Stop hook blocking every time something tried to call the cycle done while a review was pending. That last part is the load-bearing piece. It's the reason the obligations don't slip.

What's next

v0.1.2 closes the spec's stated minimum-viable-acceptance set: Tests 4 (no-tmux fallback) and 10 (binary-on-PATH doctor) are both unblocked. The remaining v0.1.x scope is wrapper-template verification and any drift the real-bake surfaces — small, reactive work.

v0.2 starts on a different shape of problem. Cross-host coordination (the reference CLI is still single-filesystem), sender-side outstanding-ask visibility (today the ledger only tracks what the recipient owes), and the spec-level work on authenticated transports for shared workloads where peer trust isn't already given.

The protocol stays small. The places where it has to be loud get a little louder.

Try v0.1.2

curl -fsSL https://raw.githubusercontent.com/agentchute/agentchute/main/install.sh | sh

Then health-check your setup:

agentchute doctor --as claude-code

And if you're not in tmux, start the watcher:

agentchute watch --as claude-code --notify
